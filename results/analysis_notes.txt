Kafka vs Sidekiq Benchmark - Analysis Notes & Action Items
25 Feb 2026

1. ALIGNED WITH LITERATURE

Kafka throughput dominance for CPU-light and bursty workloads confirmed. Kafka led cpu_light by 2.78x and fanout by 3.16x at baseline, consistent with Vyas et al., Maharjan et al., Dobbelaere & Esmaili, and Fu et al. Kafka's batched log-based fetch gave it a persistent fanout advantage (1.74x) even when Sidekiq was fully scaled out with 4 workers and 25 threads. The general finding that the best system is workload-dependent, as stated by Maharjan and Fu, is strongly confirmed across all four workloads. Topalidi's characterisation of Sidekiq as competitive for short-lived homogeneous tasks is supported by the cpu_light convergence under optimised concurrency.

2. MISALIGNED WITH LITERATURE

Maharjan et al. report Redis as having the lowest latency. Our end-to-end measurements show Kafka with lower latency in 3 of 4 workloads (cpu_light p50 882ms vs 2293ms, fanout p50 259ms vs 661ms, delayed p50 2501ms vs 6435ms). This is likely a methodological difference rather than a true contradiction - Maharjan measures broker-level latency (single message round trip) while we measure system-level latency (enqueue to job completion) which is dominated by queue depth and drain rate. See action item 5.

No cited study predicts the io_bound performance inversion where Sidekiq was 4.28x faster than Kafka at baseline. The literature generally positions Kafka as the higher throughput system across all workloads. See consideration 4.

3. UNCONFIRMED DUE TO NON-ISOLATED VARIABLES

The baseline-to-optimised comparison changes multiple parameters simultaneously (partitions 6 to 12, consumers 1 to 4, threads 1 to 25, compression none to snappy, acks all to 1, batch size 1 to 100). This means we cannot attribute performance changes to any single parameter.

Action item: run single-variable sweeps to isolate the effect of partition count, consumer thread count, and Sidekiq concurrency independently. This is needed to properly validate Vyas et al.'s claims about partitions and polling interval as first-order performance parameters.

4. IO_BOUND PERFORMANCE INVERSION - CONSIDERATIONS

At baseline Kafka consumed 17.8 msg/s vs Sidekiq 76.2 msg/s for io_bound work (50ms HTTP call per message). This looks dramatic but may be entirely explained by concurrency configuration rather than a benchmarking error.

Kafka baseline used a single-threaded consumer. Each 50ms HTTP call blocks the thread, giving a theoretical ceiling of 20 msg/s. Measured 17.8 msg/s is consistent with this ceiling plus overhead.

Sidekiq baseline used concurrency=10. Ten threads can make 10 concurrent HTTP calls, giving a theoretical ceiling of 200 msg/s. Measured 76.2 msg/s is below this ceiling, likely due to queue dispatch overhead and connection pooling.

Under optimised config (25 threads, 4 workers) both systems converge to approximately 600 msg/s, suggesting the inversion disappears when concurrency is equalised. The stub server had 32 Gunicorn workers, giving a theoretical ceiling of roughly 640 msg/s, which both systems approached.

This is probably not a benchmarking error but rather an artifact of comparing fundamentally different default concurrency models. However, it raises the question of whether the baseline comparison is fair. Kafka's single-threaded consumer is a deliberate simplicity choice, while Sidekiq's 10-thread default is a deliberate throughput choice. A fairer baseline might match effective concurrency (e.g. Kafka with 10 worker threads vs Sidekiq with concurrency=10).

Action item: re-run io_bound with matched concurrency (same effective thread count on both sides) to determine whether the inversion is purely a concurrency artifact or if there is an underlying architectural difference in how each system handles blocking I/O.

5. ACTION ITEM - BROKER-LEVEL LATENCY MEASUREMENT

Currently we only measure end-to-end latency (enqueue_time to complete_time). This conflates broker transit time with queue wait time and processing time.

To align with Maharjan et al.'s methodology and to make a methodological contribution, we should also measure broker-level latency by recording dequeue_time - enqueue_time separately. The data is already captured (processing_ms = complete_time - dequeue_time exists in our results) so we can derive broker transit time as latency_ms - processing_ms - queue_wait_time.

Action item: add a broker_latency_ms field to the results that captures just the time from message production to the moment the consumer/worker first receives it. This will let us compare directly with Maharjan's numbers and explain the apparent Redis latency discrepancy in our Discussion chapter.
